# Go Secure Coding Practice

보안 코딩 연습을 위한 시작 프로젝트입니다.

처음부터 구조를 예쁘게 나누기보다, `cmd/server/main.go` 하나에 코드를 모아 둔 상태에서
먼저 흐름을 이해하고 직접 분리 기준을 고민할 수 있게 만드는 것이 목적입니다.

지난 과제와 수업에서 설명했던 내용, 그리고 전달된 가이드 문서를 떠올리면서
어떤 기능부터 구현하고 어떤 기준으로 나눌지 스스로 판단해 보세요.

## 프로젝트 목적

- 로그인 흐름을 먼저 이해하기
- 더미 API를 실제 동작 코드로 바꿔 보기
- 게시판과 금융 기능을 단계적으로 채워 보기
- 입력 검증, 인증 확인, 권한 검사, 응답 설계를 직접 고민해 보기
- 구현 후 어떤 코드들을 묶어 리팩터링할지 판단해 보기

## 현재 상태

- 서버 코드는 현재 `cmd/server/main.go` 하나에 들어 있습니다.
- 로그인만 SQLite 조회를 사용합니다.
- 로그인 성공 시 `authorization` 쿠키와 `Authorization` 헤더용 토큰을 함께 사용할 수 있습니다.
- 세션은 DB가 아니라 메모리 맵으로 관리합니다.
- 게시판 API는 고정 더미 응답을 반환합니다.
- 금융 API는 더미 응답을 반환합니다.
- 정적 화면은 SPA 형태로 준비되어 있습니다.

## 먼저 해볼 작업

1. 회원가입 더미 핸들러를 실제 `INSERT`로 바꿔 보기
2. 게시글 작성, 조회, 수정, 삭제를 실제 SQL 또는 원하는 저장 방식으로 바꿔 보기
3. 입금, 출금, 송금 더미 핸들러를 실제 로직으로 바꿔 보기
4. 실패 조건과 입력 검증 규칙을 정리해 보기
5. 반복되는 인증 확인 코드를 어떻게 줄일지 생각해 보기

## 작업하면서 점검할 질문

- 이 코드는 요청 처리인지, 비즈니스 규칙인지, DB 접근인지?
- 같은 인증 확인 코드가 반복되고 있지 않은가?
- 게시글 수정/삭제 권한 검사는 어디에 두는 것이 자연스러운가?
- 출금과 송금에서 어떤 실패 조건을 먼저 막아야 하는가?
- 어떤 응답은 그대로 내려도 되고, 어떤 응답은 가공이 필요한가?
- 언제부터 `handler`, `service`, `store` 같은 구조로 나누는 것이 좋은가?

## 실행 방법

프로젝트 루트에서 실행합니다.

```powershell
go run ./cmd/server
```
처음 상태로 다시 시작하고 싶으면 `app.db`를 지운 뒤 다시 실행하면 됩니다.

### 작업한 내용의 특징에 대해서 작성해주세요
ex)
> main.go (20 ~ 29)
: 비밀번호 해시와 같은 민감정보가 응답(Output)으로 나가는 것을 프레임워크 레벨에서 원천 차단하기 위해 	Password string `json:"-"`을 이용하였다. 

또한 대소문자를 구분하여 캡슐화를 진행하였고, 
이름의 첫글자 대소문자로 모든 권한을 통제하려 하였다. 대문자로 시작하는 경우엔 패키지 외부에서 접근 가능하여 JSON 파싱이나 DB ORM 매핑이 가능하도록 하였고, 

소문자로 시작할 경우는 패키지 내부에서만 접근 가능하도록 하여 보안에 민감한 키나 로직을 숨길 수 있게 하였다. 

SQL injection을 방지하기 위해 Store 함수들의 전반에 사용자 입력을 직접 문자열로 붙이지 않고 `?` 바인딩 형식을 사용하였다. 

또한 newSessionToken()은 암호화를 기반으로 무작위 토큰을 생성하여 예측 불가능한 세션 토큰을 생성하였고, 인증 토큰 탈취를 방지하려는 의도를 담았다. 

현재 비밀번호는 평문으로 저장되어 있기 때문에 보안 취약점이 존재한다. 향후 암호화하는 로직 적용이 필요하다.

이체와 출금은 시간 상의 문제, 내 역량 문제로 서버 내에서 기능이 동작하도록 구현하진 못했다. 하지만 트랜잭션 처리를 위해 여러 개의 쿼리가 모두 성공하거나 하나라도 실패할 경우 모두 취소되도록 하였다. 작업 도중 에러시에 롤백함수가 자동 발동하며, 모두 성공할 때에만 명시적으로 호출 가능하도록 하였다. 

또한 게시글 관련 함수에 권한을 검증하는 로직을 추가하여 게시글 작성자 또는 관리자만 수정 및 삭제 가능하도록 만들었다. 이는 IDOR 취약점의 예방과 관련이 있다. 

ShouldBindJSON을 사용하였고, 이는 클라이언트의 복잡한 JSON 바디를 GO 구조체로 즉각 파싱할 수 있게 하였다. 이는 구조체 태그에 binding:"required" 속성을 달아두면 필드 누락 시에 에러를 뱉는다. 또하 ㄴ파라미터로 구조체의 포인터를 넘겨 메모리 공간을 프레임워크에 위임할 ㅅ우 있다는 ㅈ장점ㅇ이 존재한다. 

추가할 수 있는 코드로는 접속 로그를 생성하는 것이 있을 것 같다. OS 패키지를 이용하여 Go 서버 로컬 디스크에 직접 기록을 남기고, 접속 및 기록을 남긴 사용자가 누구인지, 회원가입을 한 사용자는 누구이고, 어떤 행위를 하였는지 로그를 남겨 보안성을 향상시킬 수 있을 것 같다. 또한 Logurs + Lumberjack 조합으로 파일에 자동 기록하여 클라이언트와 IP, 요청 경로, HTTP 메서드를 일관성있게 저장, 비즈니스 로직이 실행되기 전에 무조건적으로 로그를 남길 수 있기 때문에 관제를 보장할 수 있을 것이라 예상된다. 

개인적으로 느낀 한계와 개선점으로는 비밀번호가 평문으로 저장되어 암호화가 필요하다는 것이고, 
세션이 메모리에 저장되기 때문에 서버 재 시작 시에 로그아웃된다는 한계가 존재했다. 
또한 입금출금의  API가 아직 dummy 상태이기 때문에 기능을 전부 구현하지 못한 미완의 코드라는 한계가 존재하한다. 

입출금 API는 세션 유저의 잔액을 동기화시키는 코드와 이를 통해 현재 상태를 업데이트하는 코드를 구현하는 것이 필요한 ㄷ걱 같다. 



느낀 점으로는 최대한 AI를 사용하지 않고 코드를 짜려니 너무 어려웠고, 그나마 인터넷에 있는 자료들을 사용하여 구현하자니 Go라는 언어가 익숙하지 않고 백엔드 개발 경험이 현저히 부족했기에 기본 API 작성과 기능을 구현하는 것에도 어려움을 느껴 많이 헤메게 되었다. 내가 얼마나 AI에 의존하고 있었는지 반성하게 되었다.
